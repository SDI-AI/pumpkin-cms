# Pumpkin CMS Tenant App Base Requirements

`sample-app-2` is the first tenant-starter application for Pumpkin CMS. It starts from `sample-app`, keeps the public rendering path, and adds an embedded admin surface for tenant-owned site building.

## Product Goal

Create a Next.js tenant application that can be generated by CLI, connected to a Pumpkin CMS tenant, and immediately used as both:

- a fast public site renderer for Pumpkin Page documents
- a tenant admin site for theme, navigation, footer/header, page CRUD, and page-map workflows

This should become the default "new Pumpkin site" template: a WordPress replacement with API-first content, better performance, and tenant isolation built in.

## Current Baseline

- App path: `apps/sample-app-2`
- Public site routes fetch Pumpkin pages by slug.
- Public site now fetches the active tenant theme from `GET /api/themes/{tenantId}` with static fallback.
- Runtime theming uses `--pk-*` CSS variables plus semantic `pk-*` classes.
- Admin routes exist under `/admin`:
  - `/admin`
  - `/admin/themes`
  - `/admin/navigation`
  - `/admin/pages`
  - `/admin/page-map`
  - `/admin/settings`

## Runtime Theming Requirement

Tailwind cannot safely generate arbitrary class names that users create in the database after the app is built. The tenant theme system should therefore prefer this model:

1. Tailwind compiles a stable Pumpkin CSS layer at build time.
2. Tenant themes store runtime-safe design tokens such as colors, radius, shadows, width, spacing, and typography selections.
3. Tenant themes may select semantic class presets like `pk-hero`, `pk-card`, and `pk-button--primary`.
4. Raw Tailwind class strings should be treated as advanced/curated values, not the primary tenant theme mechanism.
5. If we want fully arbitrary utility generation at runtime later, it should be an explicit CSS generation pipeline with validation, not accidental Tailwind JIT expectations.

## Theme Document Direction

Keep the existing `Theme` model shape for compatibility:

- `header`
- `footer`
- `menu`
- `blockStyles`

Extend it with a tenant-runtime layer:

```ts
tokens?: {
  colorBackground?: string;
  colorSurface?: string;
  colorSurfaceMuted?: string;
  colorText?: string;
  colorTextMuted?: string;
  colorBorder?: string;
  colorPrimary?: string;
  colorPrimaryHover?: string;
  colorPrimaryText?: string;
  colorAccent?: string;
  radiusSm?: string;
  radiusMd?: string;
  radiusLg?: string;
  shadowSm?: string;
  shadowMd?: string;
  maxWidth?: string;
}
```

## Admin Area Requirements

- Auth:
  - Reuse Pumpkin admin JWT auth.
  - Tenant-instance admins need clear permission boundaries separate from super-admin tenant management.

- Themes:
  - Edit tokens with controls, not only JSON.
  - Preview changes before publishing.
  - Save active theme to Pumpkin API.
  - Keep JSON import/export for power users.

- Header/Footer/Menu:
  - Edit logo, site name, sticky behavior, CTA, footer description, copyright.
  - CRUD menu items and child items.
  - Reorder menu items.
  - Hide/show items without deleting them.

- Pages:
  - List pages by tenant.
  - Create/edit/delete pages.
  - Edit metadata, SEO, sitemap flags, publish state, content relationships.
  - Edit content blocks using the existing admin block editor infrastructure.

- Page Map:
  - Reuse the existing admin infrastructure:
    - ReactFlow canvas
    - Dagre auto-layout
    - hub/spoke edges
    - hub/sub-hub edges
    - related-hub dashed edges
    - orphaned page detection
    - persisted `layoutPositions`
  - Decide whether tenant admins can edit relationships from the map or only navigate to page edit forms.

## CLI Tenant Generation Requirements

The future CLI should be able to create a new tenant app instance with:

- app name
- tenant ID
- API URL
- public API key or deployment secret wiring
- initial active theme
- optional seed pages
- deployment target profile
- generated `.env.example`
- generated README with tenant-specific commands

## Clarifying Questions

1. Should `sample-app-2` embed the admin UI in every tenant app, or should the CLI optionally generate public-only and public-plus-admin variants?
2. Should tenant admins authenticate directly against Pumpkin API JWT auth, or should a tenant app proxy auth through Next.js route handlers?
3. Do tenant admins get full page CRUD, or should destructive operations like delete/unpublish be gated behind roles?
4. Should theme changes publish immediately, save as drafts, or support named theme versions with an active theme pointer?
5. Should runtime theme tokens be added to the shared .NET and TypeScript `Theme` models now, or kept as an app-local extension until the editor is proven?
6. Should the page map be editable in-place, or should relationship edits happen in the page editor with the map as navigation/diagnostics?
7. Should a new CLI tenant create a new Pumpkin API tenant record automatically, or only generate the front-end app for an existing tenant?
8. What is the intended hosting shape: one Next.js app per tenant, one multi-tenant Next.js app resolving by host, or both?
9. Do we want generated tenant apps to include the full `pumpkin-block-views` package, or should the CLI be able to eject/copy block views for customization?
10. For user-authored styling, do we want a constrained no-code token editor first, or a power-user CSS editor with validation and preview isolation?
